# Архитектура системы: нейроэкзаменатор

## 1. Общая архитектура

Канонические правила поведения Telegram-бота по отношению к студенту описаны отдельно в `docs/ai_behavior_spec.md`. Этот документ ниже фиксирует именно архитектурные слои и инварианты реализации.

Система построена по многоуровневой асинхронной архитектуре:
Telegram Bot
↓
FastAPI Backend (Webhook)
↓
Application Layer (FSM, Session Manager)
↓
Domain Layer (Evaluation, Speech)
↓
Infrastructure Layer (DB, Google Sheets, APIs)
↓
External Services (OpenAI, Whisper)


---

## 2. Основные компоненты

### 2.1 Telegram Bot (Transport Layer)

- Принимает:
  - команды (`/start`, `/new`)
  - текстовые сообщения
  - голосовые сообщения
- Передает данные в FastAPI через webhook
- Формирует пользовательские ответы по каноническим правилам поведения из `docs/ai_behavior_spec.md`

---

### 2.2 FastAPI Backend

- Центральная точка входа
- Обрабатывает webhook
- Маршрутизирует запросы
- Управляет сессиями пользователей

---

### 2.3 FSM (Finite State Machine)

Управляет сценарием экзамена:
START → LANGUAGE → REGISTRATION → ANSWERING → FINISH


Отвечает за:
- переходы между состояниями
- контроль времени экзамена
- завершение сессии

---

### 2.4 Session Manager
Управляет пользовательскими сессиями:
session = {
user_id,
discipline_id,
session_id,
state,
start_time,
language
}


Обеспечивает:
- изоляцию пользователей
- поддержку параллельных сессий
- контроль таймаута

---

### 2.5 Evaluation Service

Вызывается **для одной пары** `(question_key, фрагмент ответа студента)`: эталон подбирается по ключу вопроса в билете.

Функции:
- сопоставление ответа с эталоном
- расчет оценки
- генерация объяснения

Использует:
- coverage / rubric по смысловым элементам — основной режим
- similarity по эмбеддингам — резервный режим
- LLM (при необходимости)

---

### 2.6 Speech Service

Функции:
- преобразование аудио в текст (STT)
- нормализация текста

Использует:
- Whisper API

**Инвариант (MVP):** одно голосовое сообщение может содержать ответы на **несколько вопросов** билета. STT возвращает **единый транскрипт**; **сегментация** текста и **сопоставление фрагментов с `question_key`** выполняются на уровне Application / Domain (FSM + сервис сценария или отдельный компонент), после чего Evaluation вызывается **по каждому ключу отдельно**. Способ сегментации (маркеры, порядок вопросов, эвристики, LLM) — деталь реализации при сохранении инварианта «одно голосовое → N ответов по N ключам».

---

### 2.7 Persistence Layer

#### PostgreSQL (основное хранилище)
Хранит:
- пользователей
- сессии
- ответы
- оценки
- логи

#### Google Sheets (операционный слой)
Используется для:
- просмотра кафедрой
- редактирования эталонов
- выгрузки результатов

Образец структуры таблицы (выгрузка из Google Sheets): `docs/sample table.xlsx`.

---

### 2.8 Background Workers

Используются для:
- транскрибации
- оценки
- повторных попыток записи

Технологии:
- Redis
- Celery или RQ

## 4. Поток обработки запроса

1. Пользователь отправляет сообщение в Telegram
2. Telegram вызывает webhook FastAPI
3. Backend:
   - определяет пользователя и дисциплину
   - извлекает сессию
4. FSM определяет состояние:
   - если **голос** → Speech Service (Whisper) → **сегментация транскрипта и привязка фрагментов к `question_key`** текущего билета
   - если **текст** → привязка к ключу(ам) по правилам сценария (в простейшем случае — один ключ за сообщение)
5. Evaluation Service (для **каждой** пары ключ + текст ответа):
   - сравнивает с эталоном по этому ключу
   - рассчитывает оценку
6. Результат:
   - сохраняется в PostgreSQL
   - синхронизируется в Google Sheets
7. Ответ возвращается пользователю

## 6. Multi-discipline модель

- 1 дисциплина = 1 Google Sheet
- 3 листа (имена в проде задаются в `.env`, типично):
  - `ideal_answers` — эталоны (`GOOGLE_SHEET_IDEAL_TAB`)
  - `students_answers` — результаты (`GOOGLE_SHEET_RESULTS_TAB`)
  - metadata

Данные НЕ удаляются

Используются поля:
- group_id
- session_id
- attempt_id

---

## 7. Multi-group модель

- 1 дисциплина → несколько групп
- ~12 человек в группе
- 3–6 активных одновременно

Система должна:
- поддерживать параллельную работу
- изолировать пользователей
- группировать результаты

---

## 8. Стратегия бота

- Один бот обслуживает:
  - несколько дисциплин
  - несколько групп

Маршрутизация:

user → discipline → group → session


Отдельный бот на группу НЕ требуется

---

## 9. Нагрузочная модель

- 3–5 дисциплин
- 3–6 пользователей на дисциплину
- пик: 20–25 пользователей

Требования:
- асинхронная обработка
- неблокирующий код
- использование очередей

## 11. Порядок внедрения MVP

Цель раздела — зафиксировать **рекомендуемый порядок работ**, чтобы не держать его в голове. Это дополняет этапы из `docs/02-delivery-model.md` (процесс) конкретикой **слоёв системы**.

### 11.1 Точка входа приложения

- Живой entry point FastAPI — **`main.py` в корне репозитория** (создание `app`, подключение роутеров). Перенос в `app/main.py` не обязателен и остаётся вкусовым совпадением с черновиком дерева в разделе 3.

### 11.2 Вертикальные шаги (от тонкого к полному)

Рационально идти **сквозным срезом** (один сценарий end-to-end), наращивая сложность:

| Шаг | Что делаем | Зачем |
|-----|------------|--------|
| 1 | **Webhook Telegram** (маршрут, парсинг `Update`, секрет/валидация по документации Bot API) | Центральный транспорт сценария экзамена |
| 2 | **Минимальная сессия + FSM** (состояния, таймаут экзамена по брифу) | Изоляция пользователя и сценарий без «памяти в коде» |
| 3 | Цепочка **один ответ — один `question_key`**: существующие **Speech → Evaluation** | Проверка STT и скоринга в реальном потоке бота |
| 4 | **Одно голосовое → N ключей**: сегментация транскрипта и цикл оценки по эталонам из таблицы | Инвариант продукта (см. `docs/02-delivery-model.md`, 2A) |
| 5 | **PostgreSQL**: пользователи, сессии, попытки (`Attempt` по ключам) | Надёжность и аудит |
| 6 | **Google Sheets**: чтение эталонов / выгрузка результатов; образец колонок — `docs/sample table.xlsx` | Операционный слой для кафедры |
| 7 | **Очередь** (Redis + Celery/RQ) для долгих STT/оценок, неблокирующий webhook | Масштаб и таймауты Telegram |

Шаги 5–7 на раннем MVP можно упрощать (например, сначала без очереди), но порядок **сначала webhook и сессия, потом персистентность и Sheets** снижает переделки.

### 11.3 Секреты и Telegram

- **Токен бота** выдаётся [@BotFather](https://t.me/BotFather); задаётся только в **`.env`** (шаблон — `.env.example`). Не коммитить, не вставлять в чаты и тикеты.
- Для **webhook** нужен **публичный HTTPS**-URL приложения (туннель вроде ngrok/cloudflared или деплой). Локальный `localhost` без туннеля для webhook не подходит.
- Команды запуска для разработки (uvicorn, регистрация webhook) задаются вручную в среде разработчика; в репозитории достаточно ссылаться на `README.md` после их появления.
