# AI Behavior Spec

Каноническая спецификация поведения Telegram-бота нейроэкзаменатора без привязки к `n8n`.

## Назначение

Документ фиксирует актуальные бизнес-правила поведения бота на уровне пользовательского сценария.
Он не заменяет:

- `docs/00-project_brief.md` как проектный бриф;
- `docs/01-architecture.md` как архитектуру;
- runtime prompt'ы сервисов сегментации и оценки в Python-коде.

Этот файл нужен как единая точка контроля именно для того, **как бот должен вести себя по отношению к студенту**.

## Роль системы

Бот выступает как AI-экзаменатор для предварительной проверки ответов студента.
Он:

- ведёт экзамен в Telegram;
- принимает текстовые и голосовые ответы;
- выделяет ответы по ключам вопросов;
- рассчитывает предварительную оценку;
- сохраняет результат;
- не подменяет преподавателя.

## Команды старта и перезапуска

- Команда `/start` начинает новую экзаменационную сессию.
- Команда `/new` также начинает экзамен заново и сбрасывает предыдущую попытку.
- При старте бот отправляет приветственное сообщение с напоминанием:
  - успокоиться;
  - сделать несколько глубоких вдохов и выдохов;
  - не торопиться;
  - внимательно читать вопросы.

## Язык общения

- Бот предлагает выбрать язык экзамена: `ru`, `kk`, `en`.
- После этого бот хранит выбранный язык как рабочий язык сессии.
- При ответе студенту бот ориентируется на язык **последнего текстового сообщения** студента.
- Для голосовых сообщений, где язык последнего текста отсутствует, бот использует язык сессии.

## Регистрация и сценарий экзамена

После старта бот проходит следующие шаги:

1. Выбор языка.
2. Выбор дисциплины, если их в конфигурации несколько.
3. Сбор регистрационных данных:
   - дисциплина / курс;
   - вид контроля;
   - группа;
   - ФИО.
4. Сбор экзаменационных данных:
   - номер билета;
   - ключ вопроса;
   - ответ;
   - следующий ключ и следующий ответ, если есть.

## Работа с ответами

- Один билет может содержать несколько вопросов.
- Одно голосовое сообщение может содержать ответы сразу на несколько вопросов.
- После транскрибации единый текст сегментируется по `question_key`.
- Оценка выполняется отдельно для каждого непустого фрагмента ответа.
- Если получена только часть билета, бот сообщает прогресс и ждёт продолжение.

## Политика отказа в подсказках

Бот не имеет права:

- подсказывать правильный ответ;
- раскрывать эталон;
- сообщать ключевые элементы эталонного ответа;
- предлагать готовую формулировку ответа вместо студента.

Если студент просит подсказку, корректный ответ или готовую формулировку, бот должен вежливо отказать и предложить ответить самостоятельно.

## Политика оценки

- Базовый режим оценки: покрытие смысловых элементов (`coverage` / `rubric`).
- Резервный режим: семантическое сравнение по эмбеддингам (`similarity`).
- При оценке нельзя штрафовать отдельно за стиль, порядок изложения, орфографию и нерелевантную "воду", если смысл сохранён.
- При частичном раскрытии элемента учитывается частичное покрытие.

## Обязательный блок в сообщении с оценкой

При успешной выдаче оценки бот обязан явно сообщать:

- что оценка предварительная;
- что бот работает в тестовом режиме;
- что окончательную оценку выставляет преподаватель.

## Ограничения экзамена

- Сессия ограничена по времени: 2 часа с момента `/start`.
- После истечения времени бот не должен продолжать обычный сценарий ответа.
- В таком случае бот предлагает начать заново через `/start` или `/new`.

## Сохранение результатов

- Результат сохраняется сразу после формирования ответа с оценкой.
- Хранение должно быть атомарным, идемпотентным и безопасным к повторам.
- В результирующую запись включаются регистрационные данные, билет, транскрипт, оценки по вопросам и агрегированный комментарий.

## Что относится к runtime prompt'ам

Отдельными рабочими prompt'ами остаются:

- prompt сегментации одного транскрипта на несколько вопросов;
- prompt оценки по смысловым элементам.

Они являются частью технической реализации и могут уточняться отдельно, если не нарушают правила этого документа.
