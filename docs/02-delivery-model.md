# Этап 2. План работ и модель ведения проекта

## Назначение документа

Документ фиксирует рабочую модель проекта после подготовки:
- Этапа 0: бриф;
- Этапа 1: архитектура.

Цель этого документа — определить:
- как проект декомпозируется на этапы;
- какие артефакты ожидаются на каждом этапе;
- какие критерии готовности применяются;
- как разделяется работа по чатам, чтобы не смешивать контекст.

---

## 1. Место этапа 2 в общей схеме

### Этап 0. Бриф
**Вход:** исходная идея, бизнес-цели, ограничения, требования  
**Выход:** согласованный бриф

### Этап 1. Архитектура
**Вход:** бриф  
**Выход:** архитектурное описание, инварианты, ключевые контракты, структура компонентов

### Этап 2. План работ и модель исполнения
**Вход:** архитектура + требования  
**Выход:** план, разбитый на этапы, с артефактами, критериями готовности, зависимостями и минимальными проверками

### Этап 3. Реализация
**Формат:** отдельный чат на каждый этап реализации

### Этап 4. Документация
**Формат:** отдельный чат “Документация”

### Этап 5. DevOps
**Формат:** отдельный чат “DevOps”

### Этап 6. Security review
**Формат:** отдельный чат “Security review”

---

## 2. Принципы ведения проекта

1. Архитектура и бриф являются опорными документами и не переписываются без явного решения.
2. Реализация идёт этапами; каждый этап завершает конкретный кусок системы.
3. На каждый этап реализации создаётся отдельный чат.
4. Документация ведётся отдельно от реализации, но регулярно сверяется с кодом.
5. DevOps и security review не смешиваются с реализацией функционала.
6. Любой этап считается завершённым только при наличии артефактов и минимальной проверки.
7. Публичные контракты не должны расходиться с архитектурными инвариантами.

---

## 3. Этап 2 как результат планирования

### Цель
Преобразовать требования и архитектуру в управляемый план реализации.

### Вход
- архитектура;
- требования;
- зафиксированные инварианты;
- ограничения по технологиям, ресурсам и срокам.

### Выход
- карта этапов;
- зависимости между этапами;
- приоритеты;
- артефакты по каждому этапу;
- критерии готовности;
- минимальные проверки на завершение этапа.

### Риски
- слишком крупные этапы;
- этапы без проверяемого результата;
- смешение проектирования, реализации, документации и инфраструктуры;
- потеря связи между кодом и архитектурой;
- разрастание контекста в одном чате.

### DoD этапа 2
- для каждого этапа определены цель, вход, выход, риски;
- зависимости между этапами понятны;
- для каждого этапа определены минимальные проверки;
- отделены контуры реализации, документации, DevOps и security review.

## 4. Внутренняя структура Этапа 2

Этап 2 выполняется в одном чате “План работ”, но состоит из трёх подэтапов.

---

### 2A. Проработка домена

**Цель:** определить, что именно строится на уровне сущностей и логики

**Фокус:**
- оценка ответов;
- экзамен (сценарий);
- вопросный банк.

**Вход:**
- архитектура;
- требования;
- ограничения.

**Выход:**
- описание доменных сущностей:
  - вопрос;
  - ответ;
  - экзамен;
  - оценка;
- сценарий экзамена (поток);
- логика оценки (без кода).

**Риски:**
- поверхностная проработка логики;
- противоречие архитектуре;
- пропущенные сущности.

**Минимальные проверки:**
- все ключевые сущности определены;
- сценарий экзамена непротиворечив;
- вход/выход оценки понятен.

#### Правило: одно голосовое и несколько ключей (MVP)

- Одно входящее **голосовое сообщение** от студента может содержать ответы сразу на **несколько вопросов** билета.
- У каждого вопроса в данных — **свой ключ**; эталоны и оценка ведутся **по ключу**, а не «на всё сообщение одним блоком».
- После распознавания речи (Whisper) текст **сегментируется** и **сопоставляется с ключами**; каждый сегмент оценивается **отдельно** по эталону из таблицы для соответствующего ключа.
- Детали сегментации (явные маркеры, порядок вопросов в билете, эвристики или LLM) уточняются в архитектуре и контрактах этапа 2B, но **инвариант продукта** — «одно голосовое → N ответов по N ключам».

**Образец операционной таблицы** (структура как в Google Sheets, выгрузка в репозиторий): `docs/sample table.xlsx`.

---

### 2B. Контракты и модель данных

**Цель:** зафиксировать структуру данных и интерфейсы

**Вход:**
- результаты 2A;
- архитектурные инварианты.

**Выход:**
- модель данных (схемы);
- DTO;
- API-контракты;
- ограничения и валидации.

**Риски:**
- несогласованность API и модели;
- избыточная сложность;
- скрытые зависимости.

**Минимальные проверки:**
- соответствие архитектуре;
- согласованность DTO ↔ API ↔ данные;
- отсутствие дублирования сущностей.

---

### 2C. Декомпозиция на этапы реализации

**Цель:** разложить систему на реализуемые этапы

**Вход:**
- результаты 2A и 2B;
- ограничения проекта.

**Выход:**
- список этапов реализации;
- зависимости между этапами;
- DoD для каждого этапа;
- минимальные проверки.

**Риски:**
- этапы слишком крупные;
- зависимости неявны;
- невозможно проверить результат этапа.

**Минимальные проверки:**
- каждый этап имеет проверяемый результат;
- зависимости прозрачны;
- DoD измерим.

Ниже описаны этапы после завершения брифа, архитектуры и этапа 2.

# Этап 3. Реализация

## Цель
Пошагово реализовать систему по этапам, не смешивая контексты.

## Формат работы
- отдельный чат на каждый этап;
- в чат передаётся:
  - описание этапа;
  - ссылки на архитектурные инварианты;
  - текущее состояние репозитория;
  - ограничения для текущей итерации.

## Вход
- описание этапа;
- ссылки на архитектурные инварианты;
- текущее состояние репозитория;
- **порядок внедрения MVP по слоям** (чеклист): `docs/01-architecture.md`, раздел **«11. Порядок внедрения MVP»**.

## Выход
- код;
- тесты (минимум: unit + базовые интеграционные, где уместно);
- короткий MD-отчёт с изменениями и инструкцией проверки.

## Артефакты
- изменения в коде;
- тесты;
- файл отчёта по этапу:
  - `docs/reports/stage-X-report.md`

## Риски
- перекрёстное загрязнение контекста;
- реализация вне архитектурных ограничений;
- “временные” заглушки;
- несогласованность DTO / API / схем.

## DoD
- проект собирается;
- тесты зелёные;
- нет заглушек;
- публичные контракты согласованы с архитектурой;
- изменения описаны в MD:
  - что;
  - зачем;
  - как проверить.

## Минимальные проверки
- unit tests;
- базовый интеграционный тест;
- линтер;
- проверка контрактов.

---

# Этап 4. Документация

## Цель
Вынести документацию в отдельный контур и регулярно сверять её с кодом.

## Формат работы
- отдельный чат: **“Документация”**

## Вход
- документация по компонентам;
- репозиторий;
- текущие контракты и примеры использования.

## Выход
- актуализированная единая проектная документация.

## Артефакты
- единая структура документов в `docs`;
- актуализированные инструкции запуска и проверки;
- фиксация ключевых решений в ADR-подобной форме.

## Риски
- документация устаревает быстрее кода;
- в документации нет примеров;
- документация описывает старые контракты;
- решения приняты, но не зафиксированы.

## DoD
- документация соответствует текущим контрактам;
- есть раздел:
  - как запустить;
  - как проверить;
- зафиксированы ключевые решения и их обоснование.

## Минимальные проверки
- сверка маршрутов/API/DTO с кодом;
- сверка примеров запросов и ответов;
- сверка инструкции запуска с реальным состоянием репозитория;
- проверка, что новые архитектурные решения зафиксированы.

---

# Этап 5. DevOps

## Цель
Отдельно проработать CI/CD, окружения, деплой, миграции и эксплуатационные практики.

## Формат работы
- отдельный чат: **“DevOps”**

## Вход
- требования к деплою;
- SLO/SLA, если определены;
- инфраструктурные ограничения;
- требования к окружениям.

## Выход
- пайплайн;
- схема окружений;
- базовые практики работы с секретами, миграциями и rollback.

## Артефакты
- CI pipeline;
- конфигурация окружений;
- описание процесса миграций;
- инструкция rollback;
- правила работы с секретами.

## Риски
- расхождение dev/stage/prod;
- небезопасное хранение секретов;
- миграции без rollback-плана;
- нереплицируемая сборка.

## DoD
- описаны окружения;
- пайплайн воспроизводим;
- определены правила миграций и rollback;
- секреты вынесены из кода и локальных хаков.

## Минимальные проверки
- сборка проекта в CI;
- прогон тестов в CI;
- проверка миграций на чистой БД;
- smoke-check после деплоя;
- базовая проверка rollback-сценария.

---

# Этап 6. Security review

## Цель
Периодически проверять проект на безопасность, надёжность и устойчивость к деградациям.

## Формат работы
- отдельный чат: **“Security review”**

## Вход
- репозиторий;
- точки входа:
  - API;
  - интеграции;
  - фоновые джобы;
  - импорт/экспорт;
  - аутентификация и авторизация.

## Выход
- список проблем и рисков;
- рекомендации;
- приоритеты исправлений.

## Артефакты
- security review report;
- реестр рисков;
- список обязательных исправлений.

## Риски
- невалидированный ввод;
- утечки данных;
- слабая аутентификация/авторизация;
- отсутствие ограничений на интеграциях;
- неполное логирование ошибок;
- опасные деградации и silent failure.

## DoD
- найденные проблемы классифицированы по приоритету;
- рекомендации привязаны к конкретным точкам системы;
- зафиксированы минимальные требования по защите и наблюдаемости.

## Минимальные проверки
- проверка аутентификации и авторизации;
- проверка обработки ошибок;
- проверка логирования критических событий;
- проверка ограничений на входные данные;
- проверка секретов и конфигурации;
- проверка уязвимых интеграционных точек.
