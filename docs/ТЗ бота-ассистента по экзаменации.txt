Требования
Проект должен обеспечивать:
1. Асинхронную работу для нескольких экзаменуемых одновременно.
2. Маршрутизацию результатов по дисциплинам, группам и сессиям.
3. Поддержку трех языков: русский, казахский, английский.
4. Ответ бота на языке последнего сообщения студента; для голосового ответа без нового текста используется язык сессии.
5. Минимум лишних уточнений в сценарии.
6. Надежную, атомарную и идемпотентную запись результатов.
7. Возможность дальнейшего развития к multi-bot и аренде ботов.
8. Команды /start и /new для запуска новой попытки.
9. Запрет на подсказки, раскрытие эталонов и ключевых элементов ответа.
10. Обязательный дисклеймер в сообщении с оценкой:
   - это предварительная оценка;
   - бот работает в тестовом режиме;
   - окончательную оценку выставляет преподаватель.

Этапность
1. Основной функционал бота.
2. Расширение на несколько ботов.
3. Разделение на свои и арендные боты с верификацией оплаты.

Назначение
Бот предназначен для документирования ответов экзаменуемых и предварительной объективизации оценки.
Система реализована как Telegram-бот и Python backend.
Эталоны и результаты хранятся в Google Sheets и/или основном хранилище проекта.

Сценарий
1. Экзаменуемый запускает /start или /new.
2. Бот отправляет приветственное сообщение:
   - успокоиться;
   - сделать несколько глубоких вдохов и выдохов;
   - не торопиться;
   - внимательно читать вопросы.
3. Экзаменуемый выбирает язык.
4. Экзаменуемый отправляет регистрационные данные:
   - дисциплина;
   - вид контроля;
   - группа;
   - ФИО.
5. Экзаменуемый отправляет номер билета, ключи вопросов и ответы.
6. Ответ может быть текстовым или голосовым.
7. Одно голосовое сообщение может содержать ответы сразу на несколько вопросов.
8. Система:
   - распознает речь через Whisper;
   - сегментирует единый ответ по question_key;
   - оценивает каждый фрагмент отдельно;
   - считает среднюю предварительную оценку по билету;
   - сохраняет результат;
   - отправляет результат с обязательным дисклеймером.

Документирование
Фиксируются:
- ID сообщения
- ID студента
- Название дисциплины
- Название контроля
- Дата и время
- ФИО
- Номер билета
- Ответ
- Оценка
- Итог

Оценка
Основной режим:
- coverage / rubric по смысловым элементам

Резервный режим:
- similarity по эмбеддингам

Правила:
- оценка должна учитывать полноту и смысл ответа, а не только буквальное совпадение;
- частичное раскрытие элемента учитывается как частичное покрытие;
- орфография, порядок изложения и нерелевантная "вода" не должны сами по себе снижать оценку, если смысл сохранен;
- студенту нельзя раскрывать эталон, ключевые элементы эталона или правильную формулировку ответа.
