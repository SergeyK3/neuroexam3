# AGENTS.md

## 1. Общие правила работы

- Все ответы, комментарии, объяснения и описания должны быть **на русском языке**.
- Допускается использование английских технических терминов, но при необходимости с кратким пояснением.
- Ответы должны быть **структурированными, краткими и по делу**.
- Избегать "размытых" формулировок и общих советов.

---

## 2. Подход к разработке

Проект разрабатывается строго по этапам:

1. Project Brief (контекст проекта)
2. Architecture (архитектура)
3. Implementation Plan (план реализации)
4. Реализация (код)
5. Тестирование и стабилизация

❗ Запрещено переходить к коду без:
- утверждённого `project_brief.md`
- утверждённой архитектуры

---

## 3. Работа с кодом

- **Не изменять код**, если это не запрошено явно.
- Все изменения выполнять через:
  - Pull Request
- Изменения должны быть:
  - минимальными
  - изолированными
  - понятными

---

## 4. Архитектурные принципы

### 4.1 Асинхронность
- Использовать асинхронные подходы (FastAPI async)
- Поддерживать многопользовательскую работу

### 4.2 Разделение ответственности
- Telegram-бот = только интерфейс
- Backend = бизнес-логика
- AI-модули = отдельный слой

### 4.3 Масштабируемость
- Поддержка нескольких ботов (multi-bot)
- Поддержка нескольких групп (multi-group)
- Конфигурация вместо хардкода

---

## 5. Работа с данными

- Запись результатов должна быть:
  - атомарной
  - идемпотентной
  - с ретраями

- Нельзя допускать:
  - потери данных
  - дублирования записей

---

## 6. Контракты данных

❗ Нельзя ломать существующие поля:

- ID сообщения
- ID студента
- ФИО
- дисциплина
- билет
- ответ
- оценка
- комментарий

Изменения только через:
- версионирование
- обратную совместимость

---

## 7. Оценка ответов

- Использовать:
  - sentence-transformers
  - cosine similarity

- Подход:
  - не бинарная оценка
  - учитывать полноту ответа
  - учитывать смысл

---

## 8. Языки

Система должна поддерживать:

- русский
- казахский
- английский

Правила:
- ответ студента может быть на любом языке
- оценка возвращается:
  - на русском
  - и на языке ответа

---

## 9. Работа с задачами Codex

- Одна задача = один этап
- Не запускать несколько задач, изменяющих одни и те же файлы
- Использовать чёткие формулировки задач

---

## 10. Правила взаимодействия

Перед реализацией:

- при сложной задаче сначала предложить план
- согласовать план
- затем реализовывать

---

## 11. Ограничения

- Не раскрывать эталонные ответы студенту
- Не давать подсказки, упрощающие экзамен
- Учитывать лимиты времени экзамена

---

## 12. Критерии качества

Любое решение должно:

- быть воспроизводимым
- быть понятным
- быть расширяемым
- не ломать текущую логику

---

## 13. Приоритеты разработки

1. Надёжность (нет потери данных)
2. Параллелизм
3. Корректность оценки
4. Расширяемость
5. Удобство использования

---

## 14. Формат ответов Codex

Ответ должен содержать:

- краткое описание решения
- список изменений
- код (если требуется)
- пояснение (если необходимо)

Без:
- лишнего текста
- повторов
- “воды”

---

## 15. Общее правило

Если есть сомнения:
- уточнить
- предложить вариант
- не делать предположений без подтверждения
